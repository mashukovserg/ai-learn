## Task 1 — Зачем вообще файн-тюнить?

Представь: у тебя есть GPT-4 или Claude. Они умные, знают кучу всего. Но вот проблема — ты работаешь в юридической фирме, и тебе нужна модель, которая:

- Отвечает **только** в формате юридического заключения
- Знает внутреннюю терминологию компании
- Не галлюцинирует про законы, а цитирует конкретные статьи
- Работает быстро и дёшево (а не за $0.03/1K токенов)

Базовая модель так не умеет «из коробки». У тебя три пути:

| Подход | Суть | Когда использовать |
|--------|------|-------------------|
| **Prompt Engineering** | Пишешь подробный системный промпт | Простые задачи, прототипирование |
| **RAG** | Подключаешь внешнюю базу знаний | Нужны актуальные/приватные данные |
| **Fine-Tuning** | Дообучаешь саму модель | Нужно изменить *поведение*, стиль, формат |

Fine-tuning — это когда ты берёшь предобученную модель и **доучиваешь** её на своих данных. Не с нуля (это pre-training, и стоит миллионы долларов), а именно адаптируешь уже готовую модель под свою задачу.

**Аналогия:** Pre-training — это как получить высшее образование. Fine-tuning — это как пройти стажировку в конкретной компании. Ты уже умный, но теперь учишься работать *именно здесь*.

### ❓ Вопросы к Task 1

1. Какой подход лучше, если тебе нужно изменить **формат** ответов модели, а не добавить ей новые знания?
   <details><summary>Ответ</summary>Fine-Tuning</details>

2. Чем fine-tuning отличается от pre-training?
   <details><summary>Ответ</summary>Fine-tuning дообучает уже предобученную модель на небольшом датасете, а pre-training — это обучение модели с нуля на огромном корпусе данных</details>

3. У тебя есть база из 10 000 PDF-документов, и ты хочешь, чтобы модель отвечала на вопросы по ним. Какой подход лучше?
   <details><summary>Ответ</summary>RAG — потому что задача в доступе к данным, а не в изменении поведения модели</details>

## Task 2 — Виды файн-тюнинга

Не весь файн-тюнинг одинаков. Есть несколько ключевых подходов:

### 2.1 Full Fine-Tuning

Обновляются **все** веса модели. Это мощно, но:
- Требует много GPU (для 7B модели — минимум 1×A100 80GB)
- Риск «catastrophic forgetting» — модель может забыть то, что знала раньше
- Нужно много данных (тысячи-десятки тысяч примеров)

### 2.2 LoRA (Low-Rank Adaptation)

Ключевая идея: не трогай основные веса, а **добавь маленькие обучаемые матрицы** рядом с ними.

```
Оригинальный слой: W (размер d×d, заморожен)
LoRA-добавка:      B×A (размер d×r × r×d, где r << d)

Итого: W' = W + B×A
```

Параметр `r` (ранг) — обычно 4, 8, 16 или 32. Чем меньше — тем легче обучать, но меньше ёмкость.

**Почему это работает:** Исследования показали, что изменения весов при файн-тюнинге имеют низкий ранг. То есть модели не нужно менять ВСЁ — достаточно подкрутить относительно небольшое подпространство.

Преимущества LoRA:
- Обучаешь ~0.1–1% параметров (вместо 100%)
- Можно на одном GPU с 24GB VRAM для 7B модели
- Легко переключаться между адаптерами — у тебя одна базовая модель и несколько «насадок»

### 2.3 QLoRA

LoRA + квантизация базовой модели до 4-bit. Это позволяет файн-тюнить 7B модель на GPU с **6GB VRAM** (да, на обычной игровой карте).

```
Память:
Full FT 7B модели:  ~28 GB (fp32) или ~14 GB (fp16)
LoRA:               ~8-10 GB  
QLoRA:              ~4-6 GB   ← вот это уже RTX 3060
```

### 2.4 Другие методы (кратко)

- **Prefix Tuning** — обучаешь «виртуальные токены», которые добавляются к началу каждого промпта
- **Adapters** — вставляешь маленькие обучаемые слои *между* слоями трансформера
- **RLHF / DPO** — файн-тюнинг на основе человеческих предпочтений (так делают Anthropic и OpenAI для выравнивания моделей)

### ❓ Вопросы к Task 2

1. Какой метод позволяет файн-тюнить 7B модель на обычной игровой видеокарте?
   <details><summary>Ответ</summary>QLoRA</details>

2. Что такое ранг (r) в LoRA?
   <details><summary>Ответ</summary>Размерность низкоранговой матрицы, определяющая ёмкость адаптера. Чем больше r, тем больше параметров обучается, но и больше нужно ресурсов</details>

3. Почему при Full Fine-Tuning возникает риск catastrophic forgetting?
   <details><summary>Ответ</summary>Потому что обновляются все веса модели, и новые данные могут «перезаписать» знания, полученные при pre-training</details>

## Task 3 — Данные: самая важная часть

> «Fine-tuning is 90% data, 10% training.»

### Формат данных

Стандартный формат — **пары инструкция-ответ** (instruction tuning):

```json
[
  {
    "instruction": "Переведи на юридический язык",
    "input": "Вася не заплатил за квартиру 3 месяца",
    "output": "Гражданин Иванов В.А. допустил просрочку по оплате жилищно-коммунальных услуг сроком 3 (три) календарных месяца, что является нарушением ст. 153 ЖК РФ..."
  },
  {
    "instruction": "Классифицируй тип договора",
    "input": "Стороны договорились о поставке 100 единиц товара...",
    "output": "Договор поставки (ст. 506 ГК РФ)"
  }
]
```

Для чат-моделей используется формат **conversations**:

```json
{
  "messages": [
    {"role": "system", "content": "Ты юридический консультант..."},
    {"role": "user", "content": "Что делать, если сосед затопил?"},
    {"role": "assistant", "content": "В соответствии со ст. 1064 ГК РФ..."}
  ]
}
```

### Сколько данных нужно?

| Задача | Минимум | Оптимально |
|--------|---------|------------|
| Стиль/формат ответов | 50–100 примеров | 200–500 |
| Классификация | 100–500 на класс | 1000+ на класс |
| Генерация специализированного текста | 500–1000 | 5000–10000 |
| Обучение новым знаниям | Не рекомендуется (используй RAG) | — |

### Качество > Количество

Золотое правило: **100 идеальных примеров лучше, чем 10 000 мусорных**. Каждый пример в датасете — это сигнал для модели. Шум в данных = шум в ответах.

Чеклист качества данных:
- [ ] Ответы соответствуют тому, как ты ХОЧЕШЬ, чтобы модель отвечала
- [ ] Нет противоречивых примеров (одинаковый input → разный output)
- [ ] Разнообразие инструкций (не один и тот же вопрос 100 раз)
- [ ] Ответы полные и точные
- [ ] Формат консистентный по всему датасету

### ❓ Вопросы к Task 3

1. Почему fine-tuning — плохой способ «вложить» в модель новые знания (например, содержание корпоративной wiki)?
   <details><summary>Ответ</summary>Модель может галлюцинировать и смешивать новые знания со старыми. Для доступа к конкретным данным лучше подходит RAG, где модель ищет по базе, а не пытается «запомнить» всё</details>

2. В каком формате нужны данные для fine-tuning чат-модели?
   <details><summary>Ответ</summary>В формате conversations (messages) с ролями system/user/assistant</details>

## Task 4 — Практика: файн-тюним модель

Вот минимальный рабочий пайплайн с использованием **Hugging Face + PEFT (LoRA)**:

### Шаг 1: Установка

```bash
pip install transformers datasets peft trl accelerate bitsandbytes
```

### Шаг 2: Загрузка модели с квантизацией (QLoRA)

```python
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
import torch

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,
)

model_name = "mistralai/Mistral-7B-v0.1"

model = AutoModelForCausalLM.from_pretrained(
    model_name,
    quantization_config=bnb_config,
    device_map="auto",
)

tokenizer = AutoTokenizer.from_pretrained(model_name)
tokenizer.pad_token = tokenizer.eos_token
```

### Шаг 3: Настройка LoRA

```python
from peft import LoraConfig, get_peft_model, prepare_model_for_kbit_training

model = prepare_model_for_kbit_training(model)

lora_config = LoraConfig(
    r=16,                      # ранг — баланс между качеством и скоростью
    lora_alpha=32,             # скейлинг (обычно 2×r)
    target_modules=[           # какие слои адаптируем
        "q_proj", "k_proj", 
        "v_proj", "o_proj"
    ],
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# → trainable params: 13,631,488 || all params: 3,752,071,168 || trainable%: 0.36%
```

### Шаг 4: Подготовка данных

```python
from datasets import load_dataset

dataset = load_dataset("json", data_files="my_data.json", split="train")

def format_prompt(example):
    return f"""### Инструкция:
{example['instruction']}

### Вход:
{example['input']}

### Ответ:
{example['output']}"""

dataset = dataset.map(lambda x: {"text": format_prompt(x)})
```

### Шаг 5: Обучение

```python
from trl import SFTTrainer
from transformers import TrainingArguments

training_args = TrainingArguments(
    output_dir="./results",
    num_train_epochs=3,
    per_device_train_batch_size=4,
    gradient_accumulation_steps=4,    # эффективный batch = 4×4 = 16
    learning_rate=2e-4,
    warmup_steps=100,
    logging_steps=10,
    save_strategy="epoch",
    fp16=True,
)

trainer = SFTTrainer(
    model=model,
    train_dataset=dataset,
    args=training_args,
    dataset_text_field="text",
    max_seq_length=2048,
)

trainer.train()
```

### Шаг 6: Сохранение и использование

```python
# Сохраняем только LoRA-адаптер (несколько MB!)
model.save_pretrained("./my-legal-adapter")

# Потом загружаем обратно
from peft import PeftModel

base_model = AutoModelForCausalLM.from_pretrained(model_name, ...)
model = PeftModel.from_pretrained(base_model, "./my-legal-adapter")
```

### ❓ Вопросы к Task 4

1. Что делает параметр `lora_alpha`?
   <details><summary>Ответ</summary>Это скейлинг-фактор для LoRA-адаптера. Итоговое обновление весов = (lora_alpha / r) × B×A. Обычно ставят lora_alpha = 2×r</details>

2. Зачем нужен `gradient_accumulation_steps`?
   <details><summary>Ответ</summary>Позволяет эмулировать большой batch size при ограниченной GPU-памяти. Градиенты накапливаются за несколько шагов и применяются разом</details>

3. Почему LoRA-адаптер занимает всего несколько мегабайт?
   <details><summary>Ответ</summary>Потому что сохраняются только низкоранговые матрицы B и A, а не вся модель. При r=16 и d=4096 это 2×(4096×16) = 131K параметров на слой вместо 4096×4096 = 16.7M</details>

## Task 5 — Файн-тюнинг через API (без GPU)

Не у всех есть A100. Провайдеры предлагают fine-tuning как сервис:

### OpenAI Fine-Tuning API

```python
# Формат данных: JSONL
# {"messages": [{"role":"system","content":"..."},{"role":"user","content":"..."},{"role":"assistant","content":"..."}]}

from openai import OpenAI
client = OpenAI()

# 1. Загружаем файл
file = client.files.create(
    file=open("training_data.jsonl", "rb"),
    purpose="fine-tune"
)

# 2. Запускаем файн-тюнинг
job = client.fine_tuning.jobs.create(
    training_file=file.id,
    model="gpt-4o-mini-2024-07-18"
)

# 3. Используем модель
response = client.chat.completions.create(
    model="ft:gpt-4o-mini-2024-07-18:my-org::abc123",
    messages=[...]
)
```

## Task 6 — Оценка и подводные камни

### Как понять, что файн-тюнинг сработал?

1. **Train/Eval Loss** — должен снижаться, но без переобучения (eval loss растёт = переобучение)
2. **Ручная проверка** — дай модели 20–50 промптов, которых не было в обучении, и оцени
3. **A/B тестирование** — сравни ответы базовой и файн-тюненной модели
4. **Бенчмарки** — если есть метрики для задачи (accuracy, F1, BLEU), замерь до и после

### Типичные ошибки

**Overfitting (переобучение)**
- Симптомы: модель идеально повторяет обучающие примеры, но плохо генерализует
- Решение: больше данных, меньше эпох, dropout, меньше r

**Underfitting (недообучение)**
- Симптомы: модель мало изменилась по сравнению с базовой
- Решение: больше эпох, выше learning rate, больше r

**Catastrophic Forgetting**
- Симптомы: модель стала хороша в новой задаче, но забыла базовые навыки
- Решение: использовать LoRA (а не full FT), добавить общие примеры в датасет

**Garbage In — Garbage Out**
- Симптомы: модель выдаёт странные/непоследовательные ответы
- Решение: почисти данные. Серьёзно, почисти данные.

### ❓ Вопросы к Task 6

1. Как определить, что модель переобучена?
   <details><summary>Ответ</summary>Eval loss начинает расти при снижающемся train loss. Модель хорошо воспроизводит обучающие примеры, но плохо работает на новых данных</details>

2. Почему LoRA менее подвержена catastrophic forgetting, чем full fine-tuning?
   <details><summary>Ответ</summary>Потому что основные веса модели заморожены — LoRA обучает только маленькие дополнительные матрицы, не затрагивая базовые знания</details>

## Task 7 — Когда НЕ нужен файн-тюнинг

Файн-тюнинг — мощный инструмент, но часто **избыточный**. Вот дерево решений:

```
Нужно ли модели знать конкретные факты/документы?
├── Да → используй RAG
└── Нет
    ├── Можно ли решить задачу хорошим промптом + few-shot примерами?
    │   ├── Да → используй prompt engineering
    │   └── Нет
    │       ├── Нужно изменить стиль/формат/поведение?
    │       │   ├── Да → Fine-Tuning ✓
    │       │   └── Нет → подумай ещё раз, точно ли нужен FT
    │       └── Нужна скорость/дешевизна на inference?
    │           └── Да → Fine-tune маленькую модель
    │               (файн-тюненная 7B часто ≥ промпт-инженеренная 70B
    │                на специфических задачах)
```

### Комбинированные подходы

На практике часто используют **RAG + Fine-Tuning**:
- Fine-tuning для формата, стиля и задачи
- RAG для актуальных знаний

Или **Fine-Tuning + Prompt Engineering**:
- Fine-tuning задаёт базовое поведение
- Промпт уточняет конкретный запрос

### ❓ Финальные вопросы

1. У тебя маленькая модель (7B), и после файн-тюнинга она на специфической задаче работает лучше, чем GPT-4 с промптом. Почему это возможно?
   <details><summary>Ответ</summary>Файн-тюнинг «впечатывает» задачу в веса модели, и ей не нужно тратить контекст на инструкции. Маленькая специализированная модель может превзойти большую универсальную на узкой задаче</details>

2. Придумай сценарий, где нужны одновременно RAG и fine-tuning.
   <details><summary>Ответ</summary>Например: юридический чат-бот, который должен (1) отвечать в формате юридического заключения (fine-tuning) и (2) ссылаться на актуальные законы и судебную практику (RAG)</details>
