# Accent & Font Picker — как устроен и задача на внедрение

Спутник интерактивного инструмента [`green-accent-picker.html`](green-accent-picker.html). Часть 1 — подробно про логику пикера. Часть 2 — самодостаточная задача для следующего агента (Codex): применить выбранные пользователем акцент и шрифт к реальному сайту.

---

## Часть 1. Логика реализации пикера

### 1.1. Зачем и главный принцип

Весь сайт берёт зелёный акцент из палитры Tailwind `emerald` через utility-классы: `text-emerald-400` (≈293 использования), `bg-emerald-500`, `border-emerald-500`, `text-emerald-300` и т.д. Значит **сменить акцент = переопределить CSS-переменные `--color-emerald-200…600`**, а не править сотни классов. Ровно этот механизм уже использует светлая тема: блок `[data-theme="saas"]` в `globals.css` переопределяет `--color-emerald-*` на затемнённые значения.

Пикер **зеркалит этот механизм**: каждый вариант акцента — это целая рампа из 5 ступеней `{200,300,400,500,600}`, которая пишется в CSS-переменные `--e2…--e6`, а все акцентные элементы макета ссылаются на них. Что видишь в пикере — то и получишь на сайте, потому что модель одна и та же.

### 1.2. Достоверный макет комнаты

Статическая копия страницы комнаты (topbar, сайдбар, хлебные крошки, заголовок, бейдж сложности, обложка, заголовок главы, текст теории с `<Term>`-ссылками, панель заданий с вопросом/вариантами/Submit/прогрессом). Все акцентные места используют `var(--e*)`; поверхности — свои переменные (`--bg`, `--card`, `--border`, `--fg`, ступени `--n300…500`), которые переключаются для светлой темы.

### 1.3. Переключатель темы (Тёмная / Светлая)

Класс `.preview.light` переопределяет **поверхностные** переменные на светлые значения SaaS (`--bg:#fafafa`, `--card:#fff`, тёмный текст, инверсия нейтральной шкалы). Рампа акцента `--e*` при этом остаётся выбранной — упрощение: на реальном сайте светлая тема дополнительно **затемняет** акцент, а пикер держит одну рампу и оценивает светлую читаемость по тёмному концу рампы (см. 1.6). Это достаточно, чтобы сравнить варианты, но финальные светлые значения агент подбирает отдельно (Часть 2, шаг 2).

### 1.4. Готовые варианты (12)

Массив `PRESETS`. Каждый элемент: `{id, nm, note, star?, ramp:[c200,c300,c400,c500,c600]}` — рампа руками подобрана согласованной (светлый→тёмный). Порядок: `Emerald` (текущий, ориентир) → `Sage` (помечен `star:true`, рекомендация для тёмной темы) → приглушённое семейство (`Eucalyptus, Moss, Fern, Pine, Olive, Seafoam`) → насыщенные/яркие (`Jade, Green, Mint, Lime`). Свотч рисует градиент из ступеней 300/400/500, имя и `★` для рекомендованного; `note` — в `title` при наведении.

### 1.5. Свой цвет

`<input type="color">` → `rampFromBase(hex)`: цвет переводится в HSL (`hexToHsl`), базовый цвет становится ступенью 400, остальные 4 выводятся сдвигом светлоты (±) и лёгким масштабированием насыщенности вокруг базы, затем обратно в hex (`hslToHex`). Так из одного цвета получается связная рампа на лету.

### 1.6. Контраст (WCAG)

`lum(hex)` считает относительную яркость (sRGB → линейное пространство по формуле WCAG), `ratio(a,b)` — контраст. Показываются два числа: **основной акцент (ступень 400) на тёмной карточке `#1c1b19`** и **тёмный конец рампы (ступень 600) на белом `#ffffff`** (приближение светлой темы). Метки: `≥4.5:1` — AA ✓, `≥3:1` — только крупный текст, иначе ✕. Именно эти цифры — авторитетный ориентир при выборе.

### 1.7. Шрифты (7)

Переменная `--read-font` применяется к `.title`, `.chapter`, `.reading` (заголовок и текст теории — то, что обсуждается; UI-чром остаётся системным sans). Массив `FONTS`: `{nm, note, stack}`. Варианты: `Iowan` (serif, текущий), `Source Serif`, `Lora`, `Spectral`, `Inter`, `IBM Plex`, `System`. Веб-шрифты грузятся через `@import` с Google Fonts (нужен интернет; без него — откат на системные serif/sans в стеке). Каждая кнопка отрисована своей гарнитурой — превью прямо в кнопке.

### 1.8. Самодостаточность

Один HTML-файл, инлайновые CSS/JS, без сборки. Единственная внешняя зависимость — Google Fonts (опциональна, есть graceful fallback). Открывается двойным кликом (`file://`); в песочнице агента `file://` не рендерится — проверять логику прогоном JS в Node.

## Часть 2. Задача для следующего агента (Codex): применить выбранные акцент и шрифт

**Статус:** ожидает решения пользователя. **Триггер:** пользователь называет финальный вариант акцента (пресет или свой hex) и шрифт. Ранний фаворит из обсуждения — **Sage для тёмной темы**; финал подтверждает пользователь.

### Входные данные (заполнить при старте)
- `ACCENT` — выбранная рампа `{200,300,400,500,600}` (из пикера или сгенерированная из hex).
- `FONT` — выбранная гарнитура теории (serif оставляем как есть, либо переключаем на sans).

### Шаги

1. **Тёмная тема — акцент.** Сейчас `globals.css` **не** переопределяет emerald (сайт берёт дефолт Tailwind). Добавить в блок `@theme` (`src/app/[lang]/globals.css`) явные `--color-emerald-200/300/400/500/600` со значениями `ACCENT`. Так все `*-emerald-*` классы подхватят новый цвет.

2. **Светлая тема — акцент.** В существующем блоке `[data-theme="saas"]` (в том же файле) уже переопределены `--color-emerald-200/300/400/500/950`. Обновить их под `ACCENT`, но значениями, **затемнёнными под белый фон** (акцент как текст на белом должен давать ≥4.5:1 — свериться с колонкой «светлая» в пикере; если тёмный конец рампы не проходит, взять ещё темнее). `--color-emerald-950` остаётся светлым (это текст на зелёных кнопках — инверсия, не трогать логику).

3. **Шрифт теории — ✅ СДЕЛАНО (2026-07-15).** Сайт переведён на **IBM Plex Sans** через `next/font/google` в `src/app/[lang]/layout.tsx` (subset'ы `latin`+`cyrillic`, начертания 400–700, переменная `--font-plex-sans`), а `globals.css` направляет её в `--font-ui` и `--font-reading`. Теория теперь sans. Отдельного действия по шрифту не требуется. Ловушка при повторе: после правки `globals.css` Turbopack может закешировать старые значения переменных — лечится `rm -rf .next` + рестарт dev.

4. **Вычистить захардкоженные зелёные, которые не подхватят рампу.** Это rgba-тени/свечения emerald: `rgba(16,185,129,…)` (emerald-500) и `rgba(52,211,153,…)` (emerald-400) в тенях/glow. Встречаются в: `src/app/[lang]/page.tsx`, `src/app/[lang]/paths/page.tsx`, `src/app/[lang]/rooms/[id]/page.tsx`, `src/components/TaskScenario.tsx`, `src/components/TaskMentor.tsx`, `src/components/TaskTimeline.tsx`, `src/components/theory/AiAgentsTheory.tsx`. Заменить на значения нового акцента (или на токен-основанный подход), иначе на них останется старый зелёный. Также проверить `text-[#082018]`-подобные тёмные тексты на кнопках — сейчас кнопки используют `text-emerald-950` (инверсия), новых hardcoded быть не должно.

5. **Проверка.** `npm run check-all` — чисто. Пройти в **обеих** темах: дашборд (hero-карточка), одну комнату с теорией + `<Term>` + панель заданий, кнопки/бейджи/прогресс. Контраст акцентного текста ≥4.5:1 (или ≥3:1 для крупного) в обеих темах. Никаких регрессий на светлой теме (проверить чипы вроде career-room, где недавно чинили `text-emerald-950`→`text-emerald-900`).

### Не входит в задачу
- Не трогать emerald-классы в 40 theory-компонентах — они наследуют цвет через рампу.
- Не менять поверхностные токены (`--color-base/card/border/…`).
- Не менять UI-шрифт чрома (только текст теории, если выбран sans).

### Синхронизация документации (по завершении)
`BACKLOG.md`(+`.ru`), `PROGRESS.md`(+`.ru`) — веха о смене акцента/шрифта; при желании удалить одноразовый `green-accent-picker.html`/`.md` после внедрения. Соблюсти docs-матрицу и запрет захардкоженных hex из `AGENTS.md`.

### Ориентир
