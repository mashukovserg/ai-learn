# Как этот проект живёт на Vercel (обучающий разбор)

Этот файл объясняет **как устроено взаимодействие проекта с Vercel** — не пошаговая инструкция (она в [`DEPLOYMENT.md`](DEPLOYMENT.md)), а разбор «почему так и что под капотом». Читается сверху вниз; каждый раздел самодостаточен.

---

## 1. Что такое Vercel и его роль здесь

Vercel — облачная платформа, заточенная под Next.js. Она делает три вещи: собирает проект (`next build`), раздаёт статику через CDN и запускает серверный код Next.js как функции по требованию.

В нашей архитектуре **Vercel держит только фронтенд**. Бэкенд (FastAPI) и база (PostgreSQL) живут отдельно на Railway. Граница проходит так:

```
Браузер ──▶ Vercel (Next.js: страницы + прокси + edge) ──▶ Railway (FastAPI) ──▶ PostgreSQL
```

Ключевая идея: браузер общается **только** с доменом Vercel. Всё, что должно уйти на бэкенд, Vercel проксирует на своей стороне. Это убирает CORS и прячет адрес бэкенда от клиента (подробнее — раздел 4).

**Продакшн-домен:** `ai-learning-platform-murex.vercel.app`

---

## 2. Карта: какие файлы «про Vercel»

| Файл | Что делает на Vercel |
|------|----------------------|
| `next.config.ts` | Прокси `/api/*` → Railway через `rewrites()` |
| `src/proxy.ts` | Edge Middleware: редирект по локали (`/` → `/ru`) |
| `src/app/api/labs/compare/route.ts` | Serverless-функция на самом Vercel (вызывает Groq) |
| `.vercel/project.json` | Привязка локальной папки к проекту Vercel (для CLI) |
| Переменные окружения (в дашборде) | `BACKEND_URL`, `GROQ_API_KEY` |
| `package.json` → `build` | Команда, которую Vercel запускает при сборке |

Отдельного `vercel.json` в проекте **нет** — Vercel применяет дефолты для Next.js (это нормально; конфиг нужен только для нестандартных случаев).

## 3. Во что Next.js превращается на Vercel: четыре рантайма

Когда Vercel выполняет `next build`, каждый маршрут попадает в один из четырёх режимов. В логе сборки они помечаются значками — вот что они значат на нашем проекте:

- **○ Static** — страница отрисована один раз при сборке и лежит на CDN как готовый HTML (быстро, дёшево). Пример: статические страницы без данных пользователя.
- **● SSG (`generateStaticParams`)** — набор страниц предгенерирован по списку параметров на этапе сборки.
- **ƒ Dynamic** — страница рендерится на сервере **на каждый запрос** (серверless-функция). Наши комнаты `/[lang]/rooms/[id]` — именно такие: у них **нет** `generateStaticParams`, поэтому Next.js рендерит их динамически и решение «комната есть / `notFound()`» принимается в рантайме.
- **ƒ Proxy (Middleware)** — `src/proxy.ts`, запускается на Edge (близко к пользователю) до того, как запрос дойдёт до страницы.

> **Почему это важно на практике.** Когда мы добавили комнату `local-models-101`, старый продакшн-деплой отдавал на неё `404`. Причина не в кэше, а в том, что код маршрута `/[lang]/rooms/[id]` в **старой** сборке не знал про новую комнату в `ROOMS_METADATA` → вызывал `notFound()`. Чтобы новая комната появилась, нужен **новый деплой** с пересборкой — правкой данных в уже задеплоенном билде это не лечится.

## 4. Мост №1 — прокси `/api/*` на бэкенд (`next.config.ts`)

Самый важный кусок интеграции:

```typescript
// BACKEND_URL задаётся в env-переменных Vercel для продакшна.
// Локально откатывается на localhost:8000.
const backendUrl = process.env.BACKEND_URL ?? 'http://localhost:8000';

async rewrites() {
  return [{ source: '/api/:path*', destination: `${backendUrl}/:path*` }];
}
```

Что здесь происходит и зачем:

1. Браузер зовёт `GET /api/me` — то есть **свой же** домен Vercel.
2. Vercel по правилу `rewrites` незаметно перенаправляет запрос на `${BACKEND_URL}/me` (Railway). Это **серверное** переписывание — адрес в браузере не меняется, клиент не видит, куда реально ушёл запрос.
3. Ответ возвращается тем же путём.

Три выгоды такой схемы:
- **Нет CORS** — для браузера всё в одном источнике (origin), кросс-доменных запросов нет.
- **Адрес бэкенда скрыт** — `BACKEND_URL` живёт только на сервере, в клиентский бандл не попадает.
- **Локально работает без настройки** — переменная не задана → фолбэк на `http://localhost:8000`, где поднят FastAPI.

`rewrites` (переписывание) — не то же самое, что `redirects`: редирект меняет URL в браузере и виден пользователю, rewrite остаётся невидимым проксированием на сервере.

## 5. Мост №2 — Edge Middleware для локалей (`src/proxy.ts`)

Файл экспортирует функцию `proxy` (в Next.js 16 middleware может называться `proxy`) и объект `config` с `matcher`. Vercel запускает его как **Edge Middleware** — код на границе CDN, до рендера страницы:

```typescript
export function proxy(request) {
  // если в пути уже есть /en или /ru — пропускаем
  // иначе читаем cookie preferred-language (дефолт 'ru') и редиректим
}

export const config = {
  matcher: ['/((?!api|_next/static|_next/image|favicon.ico|.*\\..*).*)'],
};
```

Смысл `matcher` — **не трогать** `/api/*`, статику Next и файлы с расширением. Поэтому middleware не мешает прокси из раздела 4 и не редиректит картинки. Именно из-за этого middleware заход на голый `/` мгновенно уводит на `/ru`.

## 6. Мост №3 — serverless-функция на самом Vercel (`labs/compare`)

Не всё уходит на Railway. `src/app/api/labs/compare/route.ts` — это **Route Handler**, который Vercel запускает как собственную serverless-функцию. Он ходит напрямую в Groq API:

```typescript
const apiKey = process.env.GROQ_API_KEY;      // серверный секрет
const res = await fetch('https://api.groq.com/openai/v1/chat/completions', ...);
```

Важные детали:
- Ключ `GROQ_API_KEY` читается через `process.env` **на сервере** — в браузер он не попадает.
- Этот маршрут лежит под `/api/labs/compare`, но правило `rewrites` из раздела 4 его **не** перехватывает: Next.js сначала проверяет собственные route-хендлеры и только не найдя — применяет rewrite. Так один и тот же префикс `/api` частично обслуживается самим Vercel (labs), а частично проксируется на Railway (всё остальное).

## 7. Переменные окружения

| Переменная | Где | Зачем | Локальный фолбэк |
|------------|-----|-------|------------------|
| `BACKEND_URL` | Vercel → Settings → Environment Variables | Адрес Railway-бэкенда для прокси (раздел 4) | `http://localhost:8000` |
| `GROQ_API_KEY` | Vercel env | Ключ Groq для `labs/compare` (раздел 6) | не задан → эндпоинт вернёт ошибку |

Две вещи, которые легко забыть:
1. **Env применяются на следующей сборке.** Поменяли переменную в дашборде — старый деплой продолжает работать со старым значением, пока не сделаешь новый деплой (push или redeploy).
2. **Без `NEXT_PUBLIC_` префикса** переменная видна только на сервере. Обе наши переменные серверные — и это правильно, оба значения секретны.

## 8. Линковка проекта — папка `.vercel/`

```json
// .vercel/project.json
{ "projectId": "prj_…", "orgId": "team_…", "projectName": "ai-learning-platform" }
```

Эта папка появляется, когда локальную директорию **привязали** к проекту Vercel. Она нужна CLI, чтобы знать, куда деплоить. Правила:
- **Не коммитить** — папка в `.gitignore` (строка `.vercel`). Это машинно-специфичная привязка, а `orgId`/`projectId` не стоит светить в репозитории.
- Если папки нет, CLI при первом `vercel` спросит, к какому проекту привязаться.

## 9. Два способа выкатки на продакшн

Это ключевой раздел — у Vercel два независимых пути деплоя, и важно понимать разницу.

### Способ A — Git-интеграция (авто-деплой на push)

Vercel подключается к GitHub-репозиторию и **сам** собирает новый деплой на каждый push в продакшн-ветку (`main`). Настраивается в **Settings → Git → Connect Git Repository** (нужно один раз установить GitHub-приложение Vercel и выбрать репозиторий).

- Плюс: запушил — через пару минут само выехало. Идеально для частых изменений.
- **Статус в этом проекте на 2026-05-16: НЕ подключено.** Поэтому пуш в `main` сам по себе деплой не запускает — это видно по вкладке Deployments: там нет строк для новых коммитов.

### Способ B — CLI (ручной деплой)

Требует установленного `vercel` CLI и привязки из раздела 8. Именно так выехала комната `local-models-101`:

```bash
cd ~/Desktop/projects/ai-learning-platform
vercel login        # разовая авторизация (токен сохраняется локально)
vercel --prod       # собрать и выкатить в продакшн
```

Что делает `vercel --prod` под капотом:
1. Пакует текущую **локальную** папку (не коммит из Git — важно: деплоится то, что лежит в директории).
2. Собирает на серверах Vercel (`next build`).
3. Публикует деплой на случайный URL вида `…-8s2o6w66x-….vercel.app`.
4. **Алиасит** этот деплой на продакшн-домен `ai-learning-platform-murex.vercel.app` (см. раздел 10).

Токен после `vercel login` хранится локально, повторный логин не нужен, пока он не протухнет (тогда `vercel whoami` вернёт ошибку — надо снова `vercel login`).

> **Рекомендация:** для регулярной работы стоит подключить способ A (Git), чтобы деплой шёл автоматически на каждый push. Способ B остаётся как ручной запасной вариант и для срочных выкаток.

## 10. Домены и алиасы

Каждый деплой получает **уникальный неизменяемый URL** (`…-8s2o6w66x-….vercel.app`) — по нему всегда доступна именно эта версия. Продакшн-домен (`…-murex.vercel.app`) — это **алиас**, который указывает на «текущую боевую» версию. Деплой становится продакшном, когда домен переставляют на него (CLI делает это автоматически при `--prod`).

Отсюда полезное свойство: **Instant Rollback** в дашборде — это просто переустановка алиаса на предыдущий готовый деплой, без пересборки.

## 11. Как проверить и отладить деплой

```bash
# Статус страницы на проде (200 = ок, 404 = маршрут/данные не в этой сборке)
curl -s -o /dev/null -w "%{http_code}" https://ai-learning-platform-murex.vercel.app/ru/rooms/local-models-101

# Проверить, что прокси на бэкенд жив
curl https://ai-learning-platform-murex.vercel.app/api/    # → {"status":"OK"}

# Кто я в CLI (протух ли токен)
vercel whoami

# Логи конкретного деплоя
vercel inspect <deployment-url> --logs
```

Заголовки ответа тоже подсказывают: `x-matched-path: /[lang]/rooms/[id]` значит, что маршрут сматчился (значит `404` — это логика `notFound()`, а не «маршрута нет»); `x-vercel-cache: MISS/HIT` показывает попадание в CDN-кэш.

В дашборде: вкладка **Deployments** — список всех сборок со статусами (Building / Ready / Error). Упавшую (Error) открываешь → **Build Logs** → последние строки с ошибкой.

## 12. Частые грабли (проверено на практике)

| Симптом | Причина | Лечение |
|---------|---------|---------|
| Новая комната/страница отдаёт `404` на проде | Работает старый деплой; его код не знает про новые данные (раздел 3) | Сделать новый деплой (`vercel --prod` или push при подключённом Git) |
| Push в `main` ничего не деплоит | Git-интеграция не подключена (раздел 9A) | Подключить Git **или** деплоить через CLI |
| `vercel whoami` → ошибка токена | Токен CLI протух | `vercel login` заново |
| Поменял env, а поведение старое | Env применяются только на новой сборке (раздел 7) | Redeploy / новый push |
| `/api/*` не отвечает на проде | Не задан `BACKEND_URL` или бэкенд лежит | Проверить env в Vercel и живость Railway |

## 13. Мини-глоссарий

- **Rewrite (переписывание)** — серверное перенаправление запроса без смены URL в браузере (наш `/api/*` → Railway).
- **Edge Middleware** — код, исполняемый на границе CDN до рендера страницы (наш `proxy.ts`).
- **Route Handler** — файл `route.ts` в `app/api/*`, становится serverless-функцией на Vercel (наш `labs/compare`).
- **SSG / Dynamic** — предгенерация на сборке против рендера на каждый запрос.
- **Alias** — привязка домена (например, продакшн-домена) к конкретному деплою.
- **Instant Rollback** — переключение алиаса на прошлый деплой без пересборки.
